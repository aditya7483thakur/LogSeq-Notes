# LRU Cache

## Requirements & Clarifications

- Fixed capacity cache
- Supports:
    - `get(key)` → return value or `1`
    - `put(key, value)`
- On capacity overflow:
    - Evict **Least Recently Used (LRU)** item
- Both operations must be **O(1)**
- In-memory, single-threaded
- No use of built-in LRU libraries

---

## Core Entities & Responsibilities

### Node

- Represents a single cache entry
- Holds:
    - `key`
    - `value`
    - `prev`
    - `next`
- Needed so evicted node’s key can be removed from map

---

### DoublyLinkedList

- Maintains **usage order**
- Does **not** know about capacity or HashMap
- Responsible only for ordering operations

**Key methods**

- `addToFront(node)` → mark node as MRU
- `removeNode(node)` → remove any node in O(1)
- `moveToFront(node)` → remove + add to front
- `removeLRU()` → remove and return LRU node

> Convention used: **Head = MRU, Tail = LRU**
> 

---

### LRU Cache (Orchestrator)

- Public API of the cache
- Coordinates HashMap and DoublyLinkedList
- Enforces capacity and eviction rules

**Internal state**

- `capacity`
- `HashMap<key, Node>`
- `DoublyLinkedList`

**Public methods**

- `get(key)`
- `put(key, value)`

---

## Key Design Decisions

- Use **HashMap + DoublyLinkedList** to achieve O(1)
- HashMap stores **key → Node reference**
- DoublyLinkedList handles only ordering, not eviction decisions
- LRUCache owns:
    - capacity checks
    - eviction logic
- `get` is **not read-only** → updates recency
- Clear separation of responsibilities to avoid God classes

---

## Trade-offs & Extension Scenarios

- **Concurrency**: add synchronization around get/put if needed
- **Different eviction policy**: replace list logic (e.g., LFU)
- **Persistence**: snapshot cache state if restart recovery is required
- **Large capacity**: memory usage increases linearly with capacity

---

## Pattern Touchpoints

- **Composition over inheritance**
- **Facade-like role**: LRUCache exposes simple API, hides internals
- Clean SRP:
    - Node → data
    - List → order
    - Cache → rules & coordination

---

## Complexity

- **Time**
    - `get` → O(1)
    - `put` → O(1)
- **Space**
    - O(capacity)

## Patterns used implicitly

## 1️⃣ Facade-like Pattern

`LRUCache` acts as a facade.

It hides:

- HashMap
- DoublyLinkedList
- Node mechanics

Public API is simple:

```tsx
get()put()
```

🗣 Say:

> “LRUCache acts like a facade exposing a simple API while hiding internal complexity.”
> 

---

## 2️⃣ Composition Over Inheritance

We composed:

- LRUCache → has HashMap
- LRUCache → has DoublyLinkedList

No inheritance trees.

This is a strong design choice.

---

## 3️⃣ Strategy Pattern (Optional Extension)

If we abstract eviction policy:

```tsx
interfaceEvictionPolicy {onGet()onPut()
}
```

Then LRU becomes one strategy.

But in current design:

- We did NOT explicitly implement strategy
- Just clean separation

So don’t overclaim.

---

## 4️⃣ SRP (Very Important)

| Class | Responsibility |
| --- | --- |
| Node | Data container |
| DoublyLinkedList | Order maintenance |
| LRUCache | Cache rules & coordination |

Interviewers love this clarity.

# 🎤 How to Explain Concurrency Handling in LRU (Interview Script)

Start like this:

> “Since both `get` and `put` modify the internal state of the cache — specifically the HashMap and the doubly linked list — the implementation is not thread-safe by default.”
> 

Then continue:

> “To handle concurrency, I would protect the critical section using synchronization. I’d wrap the entire `get` and `put` operations inside a lock so that updates to the map and linked list happen atomically.”
> 

Then add:

> “This ensures we don’t get inconsistent states like broken list pointers or mismatched map entries.”
> 

If they ask about performance:

> “Initially, I’d use coarse-grained locking for simplicity. If contention becomes an issue, we could move to finer-grained locking or sharding the cache.”
> 

That’s it. Stop there.
