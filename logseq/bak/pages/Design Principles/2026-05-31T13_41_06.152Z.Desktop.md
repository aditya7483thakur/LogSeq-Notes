## DRY – Don’t Repeat Yourself
collapsed:: true
	- ## What it is (interview language)
	- **Avoid duplicating logic**
	- One source of truth for a piece of behavior
	  
	  > “If logic changes, it should change in one place.”
	  > 
	  
	  ---
	- ## Why it matters in LLD
	- Reduces bugs
	- Easier to maintain
	- Cleaner designs
	  
	  ---
	- ## Small example (TypeScript)
	- ### ❌ Violates DRY
	  
	  ```tsx
	  functioncalculateDiscount(price:number):number {
	  return price *0.1;
	  }
	  
	  functioncalculateTax(price:number):number {
	  return price *0.1;// duplicate logic
	  }
	  
	  ```
	- ### ✅ Follows DRY
	  
	  ```tsx
	  functioncalculateRate(price:number,rate:number):number {
	  return price * rate;
	  }
	  
	  ```
	  
	  ---
	- ## LLD context
	- Shared validation logic
	- Common pricing rules
	- Common state transitions
- ## KISS – Keep It Simple, Stupid
  collapsed:: true
	- ## What it is (interview language)
	- Prefer **simple designs** over clever ones
	- Solve the **current problem**, clearly
	  
	  > “Simple code is easier to understand and maintain.”
	  > 
	  
	  ---
	- ## Why it matters in LLD
	- Interviewers penalize over-engineering
	- Simple designs are easier to explain
	- Reduces unnecessary abstractions
	  
	  ---
	- ## Example (conceptual)
	- ### ❌ Over-engineered
	- 5 interfaces
	- Factory + Builder
	- Only 1 requirement
	- ### ✅ KISS
	- 2–3 clear classes
	- Direct relationships
	- Easy to explain in 60 seconds
	  
	  ---
	- ## LLD context
	- Parking Lot
	- Cache
	- Rate Limiter
	  
	  Interviewers prefer:
	  
	  > “Basic working design” over “fancy architecture”
	  >
- ## YAGNI – You Aren’t Gonna Need It
  collapsed:: true
	- ## What it is (interview language)
	- **Don’t add features until they’re required**
	- Avoid “future-proofing” prematurely
	  
	  > “Build what’s asked — nothing more.”
	  > 
	  
	  ---
	- ## Why it matters in LLD
	- Extra code = extra bugs
	- Makes design harder to reason about
	- Interview time is limited
	  
	  ---
	- ## Example
	- ### ❌ Violates YAGNI
	  
	  ```tsx
	  classPaymentService {
	  pay() {}
	  refund() {}// not required
	  schedulePayment() {}// not required
	  }
	  
	  ```
	- ### ✅ Follows YAGNI
	  
	  ```tsx
	  classPaymentService {
	  pay() {}
	  }
	  
	  ```
	  
	  ---
	- ## LLD context
	- Don’t add:
	- unused strategies
	- extra enums
	- speculative features
- ## LoD – Law of Demeter
   
   We’ll use **one simple domain** throughout:
   
   👉 **Order → Customer → Address**
   
   ---
	- ## ❌ BAD DESIGN (LoD VIOLATION)
	- ### Step 1: Classes (with exposed internals)
	  
	  ```tsx
	  classAddress {
	  constructor(privatecity:string) {}
	  
	  getCity():string {
	  returnthis.city;
	  }
	  }
	  
	  classCustomer {
	  constructor(privateaddress:Address) {}
	  
	  getAddress():Address {
	  returnthis.address;// ❌ exposing internal object
	  }
	  }
	  
	  classOrder {
	  constructor(privatecustomer:Customer) {}
	  
	  getCustomer():Customer {
	  returnthis.customer;// ❌ exposing internal object
	  }
	  }
	  
	  ```
	  
	  ---
	- ### Step 2: Usage (THIS is the problem)
	  
	  ```tsx
	  const address =newAddress("Delhi");
	  const customer =newCustomer(address);
	  const order =newOrder(customer);
	  
	  // ❌ Law of Demeter violation
	  const city = order.getCustomer().getAddress().getCity();
	  console.log(city);
	  
	  ```
	  
	  ---
	- ### 🚨 What exactly went wrong here?
	  
	  Let’s trace the chain:
	  
	  ```tsx
	  order
	  .getCustomer()// Order exposes Customer
	  .getAddress()// Customer exposes Address
	  .getCity();// Address exposes City
	  
	  ```
	  
	  👉 The **caller knows too much**:
	- Order’s internals
	- Customer’s internals
	- Address’s internals
	  
	  This is what we mean by:
	  
	  > “Internal objects were showing / leaking”
	  > 
	  
	  ---
	- ### Why this is BAD (intuitive reason)
	  
	  If tomorrow:
	- Customer stops having Address
	- Address changes structure
	- City logic changes
	  
	  👉 **This calling code breaks**, even though Order didn’t change.
	  
	  That’s **tight coupling**.
	  
	  ---
	- ## ✅ GOOD DESIGN (FOLLOWS LoD)
	  
	  Now we’ll fix it properly.
	  
	  ---
	- ## Step 1: Same internal structure (NO big change)
	  
	  ```tsx
	  classAddress {
	  constructor(privatecity:string) {}
	  
	  getCity():string {
	  returnthis.city;
	  }
	  }
	  
	  ```
	  
	  ```tsx
	  classCustomer {
	  constructor(privateaddress:Address) {}
	  
	  // ✅ hides Address
	  getCity():string {
	  returnthis.address.getCity();
	  }
	  }
	  
	  ```
	  
	  ```tsx
	  classOrder {
	  constructor(privatecustomer:Customer) {}
	  
	  // ✅ hides Customer internals
	  getCustomerCity():string {
	  returnthis.customer.getCity();
	  }
	  }
	  
	  ```
	  
	  ---
	- ## Step 2: Usage (CLEAN)
	  
	  ```tsx
	  const address =newAddress("Delhi");
	  const customer =newCustomer(address);
	  const order =newOrder(customer);
	  
	  // ✅ Law of Demeter followed
	  const city = order.getCustomerCity();
	  console.log(city);
	  
	  ```
	  
	  ---
	- ## 🔑 What changed conceptually?
	- ### BAD version
	  
	  Caller says:
	  
	  > “Give me customer → give me address → give me city”
	  >
	- ### GOOD version
	   
	   Caller says:
	   
	   > “Order, give me the customer city”
	   > 
	   
	   👉 The caller:
		- Talks to **ONE object only**
		- Does not know about internals
		- Is protected from changes
		  
		  ---
		- ## ONE LINE THAT MAKES LoD CLICK 🧠
		  
		  > If I have to navigate more than one dot (.), I’m probably breaking LoD.
		  > 
		  
		  Example smell:
		  
		  ```tsx
		  a.b.c.d()
		  
		  ```
		  
		  ---
		- ## LoD vs Encapsulation (quick clarity)
		- **Encapsulation** → hides data inside a class
		- **LoD** → hides navigation between classes
		  
		  They work **together**, not separately.
		  
		  ---
		- ## Interview-safe explanation (USE THIS)
		  
		  If interviewer asks:
		  
		  > “What is Law of Demeter?”
		  > 
		  
		  Say:
		  
		  > “It means a class should interact only with its direct collaborators and not reach into the internals of other objects. This reduces coupling and makes the design safer.”
		  > 
		  
		  Perfect. Stop there.
- ## SRP – Single Responsibility Principle
  collapsed:: true
	- ## What it is (interview language)
	- A class should have **only one reason to change**
	- Each class should focus on **one responsibility**
	  
	  👉 In simple words:
	  
	  > “One class = one job”
	  > 
	  
	  ---
	- ## Why interviewers care
	- Prevents **God classes**
	- Makes code easier to change and test
	- Directly impacts **LLD quality**
	  
	  Most bad LLD answers fail **SRP first**.
	  
	  ---
	- ## ❌ Bad Example (Violates SRP)
	- ### Problem: User management
	  
	  ```tsx
	  class UserService {
	  createUser(name: string) {
	    console.log("User created:", name);
	  }
	  
	  sendEmail(name: string) {
	    console.log("Email sent to:", name);
	  }
	  
	  saveToDB(name: string) {
	    console.log("User saved to DB:", name);
	  }
	  }
	  
	  ```
	- ### Why this is bad
	- User creation logic
	- Email logic
	- Database logic
	    
	    👉 **3 reasons to change → SRP violation**
	    
	  
	  Interviewers immediately spot this.
	  
	  ---
	- ## ✅ Good Example (Follows SRP)
	- ### Step 1️⃣ Separate responsibilities
	  
	  ```tsx
	  class UserService {
	  createUser(name: string) {
	    console.log("User created:", name);
	  }
	  }
	  
	  class EmailService {
	  sendEmail(name: string) {
	    console.log("Email sent to:", name);
	  }
	  }
	  
	  class UserRepository {
	  save(name: string) {
	    console.log("User saved to DB:", name);
	  }
	  }
	  
	  ```
	- ### Step 2️⃣ Usage (orchestrated cleanly)
	  
	  ```tsx
	  const userService = new UserService();
	  const emailService = new EmailService();
	  const userRepo = new UserRepository();
	  
	  userService.createUser("Aditya");
	  userRepo.save("Aditya");
	  emailService.sendEmail("Aditya");
	  
	  ```
	- ### Why this is good
	- Each class has **one responsibility**
	- Changes are isolated
	- Easy to test and extend
	  
	  ---
	- ## How SRP appears in LLD problems
	- ### Parking Lot
	- `ParkingLot` → coordination
	- `Slot` → availability
	- `Ticket` → entry info
	- `PricingStrategy` → fee calculation
	- ### Rate Limiter
	- Limiter → decision logic
	- Storage → counters
	- Strategy → algorithm
	  
	  Interview phrase:
	  
	  > “I separated responsibilities to avoid SRP violation.”
	  > 
	  
	  ---
	- ## Common fresher mistake 🚫
	- One big `Service` class doing everything
	- Adding methods instead of creating new classes
	  
	  Interviewers think:
	  
	  > “This will become unmaintainable.”
	  >
- ## OCP – Open/Closed Principle
  collapsed:: true
	- ## What it is (interview language)
	- Software entities should be **open for extension**
	- But **closed for modification**
	  
	  👉 In simple words:
	  
	  > “Add new behavior without changing existing code”
	  > 
	  
	  ---
	- ## Why interviewers care
	- Real systems evolve constantly
	- Changing existing code increases **bug risk**
	- OCP shows you design for **future changes**
	  
	  Most good LLD answers **implicitly follow OCP**.
	  
	  ---
	- ## ❌ Bad Example (Violates OCP)
	- ### Problem: Payment processing
	  
	  ```tsx
	  class PaymentService {
	  process(type: string, amount: number): void {
	    if (type === "CARD") {
	      console.log("Paying by Card:", amount);
	    } else if (type === "UPI") {
	      console.log("Paying by UPI:", amount);
	    }
	  }
	  }
	  
	  ```
	- ### Why this is bad
	- Adding PayPal → modify this class
	- `if-else` keeps growing
	- High risk of breaking existing logic
	  
	  🚨 Interviewer reaction:
	  
	  > “This won’t scale.”
	  > 
	  
	  ---
	- ## ✅ Good Example (Follows OCP)
	- ### Step 1️⃣ Define abstraction
	  
	  ```tsx
	  interface PaymentMethod {
	  pay(amount: number): void;
	  }
	  
	  ```
	  
	  ---
	- ### Step 2️⃣ Extend behavior via new classes
	  
	  ```tsx
	  class CardPayment implements PaymentMethod {
	  pay(amount: number): void {
	    console.log("Paying by Card:", amount);
	  }
	  }
	  
	  class UpiPayment implements PaymentMethod {
	  pay(amount: number): void {
	    console.log("Paying by UPI:", amount);
	  }
	  }
	  
	  ```
	  
	  ---
	- ### Step 3️⃣ High-level code stays unchanged
	  
	  ```tsx
	  class PaymentService {
	  constructor(private paymentMethod: PaymentMethod) {}
	  
	  process(amount: number): void {
	    this.paymentMethod.pay(amount);
	  }
	  }
	  
	  ```
	  
	  ---
	- ### Usage
	  
	  ```tsx
	  const cardService = new PaymentService(new CardPayment());
	  cardService.process(100);
	  
	  const upiService = new PaymentService(new UpiPayment());
	  upiService.process(200);
	  
	  ```
	- ### Why this follows OCP
	- New payment type → **new class only**
	- No modification to `PaymentService`
	- Existing code remains stable
	  
	  ---
	- ## How OCP appears in LLD problems
	- ### Parking Lot
	- New pricing rule → new `PricingStrategy`
	- No change to core parking logic
	- ### LRU Cache
	- New eviction policy → new `EvictionPolicy`
	- Cache code remains same
	- ### Rate Limiter
	- New algorithm → new `RateLimitStrategy`
	  
	  Interview phrase:
	  
	  > “I used abstraction so new behavior can be added without modifying existing classes.”
	  >
- ## LSP – Liskov Substitution Principle
  collapsed:: true
	- ## What it is (interview language)
	- A **child class should be usable wherever its parent is expected**
	- Replacing a base class with a subclass **should not break behavior**
	  
	  👉 In simple words:
	  
	  > “Subclasses should behave like their parent promises.”
	  > 
	  
	  ---
	- ## Why interviewers care
	- Prevents **surprising behavior**
	- Ensures inheritance is used **correctly**
	- Directly affects **runtime bugs**
	  
	  Most LSP violations come from **wrong inheritance**.
	  
	  ---
	- ## ❌ Bad Example (Violates LSP)
	- ### Problem: Rectangle & Square
	  
	  ```tsx
	  class Rectangle {
	  constructor(
	    protected width: number,
	    protected height: number
	  ) {}
	  
	  setWidth(w: number) {
	    this.width = w;
	  }
	  
	  setHeight(h: number) {
	    this.height = h;
	  }
	  
	  area(): number {
	    return this.width * this.height;
	  }
	  }
	  
	  class Square extends Rectangle {
	  setWidth(w: number) {
	    this.width = w;
	    this.height = w;
	  }
	  
	  setHeight(h: number) {
	    this.width = h;
	    this.height = h;
	  }
	  }
	  
	  ```
	- ### Usage
	  
	  ```tsx
	  function calculateArea(rect: Rectangle) {
	  rect.setWidth(10);
	  rect.setHeight(5);
	  return rect.area();
	  }
	  
	  calculateArea(new Square()); // ❌ unexpected result
	  
	  ```
	- ### Why this violates LSP
	- `Square` **changes expected behavior**
	- Code using `Rectangle` breaks when replaced with `Square`
	- Substitution fails
	  
	  🚨 Interviewer thought:
	  
	  > “Inheritance used incorrectly.”
	  > 
	  
	  ---
	- ## ✅ Good Design (Fixing LSP)
	- ### Option 1️⃣: Use Abstraction properly
	  
	  ```tsx
	  interface Shape {
	  area(): number;
	  }
	  
	  class Rectangle implements Shape {
	  constructor(private w: number, private h: number) {}
	  
	  area(): number {
	    return this.w * this.h;
	  }
	  }
	  
	  class Square implements Shape {
	  constructor(private side: number) {}
	  
	  area(): number {
	    return this.side * this.side;
	  }
	  }
	  
	  ```
	- ### Usage
	  
	  ```tsx
	  const shapes: Shape[] = [
	  new Rectangle(10, 5),
	  new Square(4)
	  ];
	  
	  shapes.forEach(s => console.log(s.area()));
	  
	  ```
	- ### Why this follows LSP
	- No broken assumptions
	- No forced inheritance
	- Each class honors its contract
	  
	  ---
	- ## How LSP appears in LLD problems
	- ### Parking Lot
	- `BikeSlot` should not break rules expected from `ParkingSlot`
	- ### Payment
	- `CardPayment` should fully behave like `PaymentMethod`
	- ### Cache
	- `LRUEvictionPolicy` should not violate `EvictionPolicy` expectations
	  
	  Interview phrase:
	  
	  > “I ensured subclasses don’t change the expected behavior of the base class.”
	  >
- ## ISP – Interface Segregation Principle
  collapsed:: true
	- ## What it is (interview language)
	- **Clients should not be forced to depend on methods they don’t use**
	- Prefer **small, focused interfaces** over large ones
	  
	  👉 In simple words:
	  
	  > “Many small interfaces are better than one fat interface.”
	  > 
	  
	  ---
	- ## Why interviewers care
	- Reduces unnecessary coupling
	- Prevents dummy / empty method implementations
	- Makes systems easier to extend and test
	  
	  ISP violations are easy to spot in LLD.
	  
	  ---
	- ## ❌ Bad Example (Violates ISP)
	- ### Problem: One large interface for everything
	  
	  ```tsx
	  interface Worker {
	  work(): void;
	  eat(): void;
	  }
	  
	  ```
	  
	  ```tsx
	  class HumanWorker implements Worker {
	  work(): void {
	    console.log("Human working");
	  }
	  eat(): void {
	    console.log("Human eating");
	  }
	  }
	  
	  class RobotWorker implements Worker {
	  work(): void {
	    console.log("Robot working");
	  }
	  eat(): void {
	    // ❌ meaningless
	    throw new Error("Robots don't eat");
	  }
	  }
	  
	  ```
	- ### Why this is bad
	- `RobotWorker` is forced to implement `eat()`
	- Interface is **too broad**
	- Clients depend on methods they don’t need
	  
	  🚨 Interviewer thought:
	  
	  > “This interface is badly designed.”
	  > 
	  
	  ---
	- ## ✅ Good Example (Follows ISP)
	- ### Step 1️⃣ Split the interface by responsibility
	  
	  ```tsx
	  interface Workable {
	  work(): void;
	  }
	  
	  interface Eatable {
	  eat(): void;
	  }
	  
	  ```
	  
	  ---
	- ### Step 2️⃣ Implement only what is needed
	  
	  ```tsx
	  class HumanWorker implements Workable, Eatable {
	  work(): void {
	    console.log("Human working");
	  }
	  eat(): void {
	    console.log("Human eating");
	  }
	  }
	  
	  class RobotWorker implements Workable {
	  work(): void {
	    console.log("Robot working");
	  }
	  }
	  
	  ```
	- ### Usage
	  
	  ```tsx
	  const human = new HumanWorker();
	  human.work();
	  human.eat();
	  
	  const robot = new RobotWorker();
	  robot.work();
	  
	  ```
	- ### Why this follows ISP
	- Classes implement **only relevant interfaces**
	- No dummy methods
	- Clean separation of concerns
	  
	  ---
	- ## How ISP appears in LLD problems
	- ### Parking Lot
	- `Payable`
	- `Parkable`
	- `Chargeable`
	  
	  Instead of one huge `ParkingService` interface.
	  
	  ---
	- ### File System
	- `Readable`
	- `Writable`
	- `Deletable`
	  
	  Instead of one `FileOperations` interface.
	  
	  ---
	- ### Rate Limiter
	- `RateLimiter`
	- `RateLimitStorage`
	  
	  Each client depends only on what it uses.
	  
	  Interview phrase:
	  
	  > “I split interfaces so classes don’t depend on unnecessary methods.”
	  >
- ## DIP – Dependency Inversion Principle
  collapsed:: true
	- ## What it is (interview language)
	- **High-level modules should not depend on low-level modules**
	- **Both should depend on abstractions**
	  
	  👉 In simple words:
	  
	  > “Depend on interfaces, not concrete classes.”
	  > 
	  
	  ---
	- ## Why interviewers care
	- Reduces tight coupling
	- Makes code testable and extensible
	- Prevents ripple effects when implementations change
	  
	  Most clean LLD answers **implicitly follow DIP**.
	  
	  ---
	- ## ❌ Bad Example (Violates DIP)
	- ### Problem: Order processing tightly coupled to payment
	  
	  ```tsx
	  classCardPayment {
	  pay(amount:number):void {
	  console.log("Paying by Card:", amount);
	  }
	  }
	  
	  classOrderService {
	  private payment =newCardPayment();// ❌ concrete dependency
	  
	  checkout(amount:number):void {
	  this.payment.pay(amount);
	  }
	  }
	  
	  ```
	- ### Why this is bad
	- `OrderService` depends on `CardPayment`
	- Changing payment method → modify `OrderService`
	- Hard to test / extend
	  
	  🚨 Interviewer thought:
	  
	  > “High-level logic is tied to low-level details.”
	  > 
	  
	  ---
	- ## ✅ Good Example (Follows DIP)
	- ### Step 1️⃣ Introduce abstraction
	  
	  ```tsx
	  interfacePaymentMethod {
	  pay(amount:number):void;
	  }
	  
	  ```
	  
	  ---
	- ### Step 2️⃣ Low-level modules depend on abstraction
	  
	  ```tsx
	  classCardPaymentimplementsPaymentMethod {
	  pay(amount:number):void {
	  console.log("Paying by Card:", amount);
	  }
	  }
	  
	  classUpiPaymentimplementsPaymentMethod {
	  pay(amount:number):void {
	  console.log("Paying by UPI:", amount);
	  }
	  }
	  
	  ```
	  
	  ---
	- ### Step 3️⃣ High-level module depends on abstraction
	  
	  ```tsx
	  classOrderService {
	  constructor(privatepaymentMethod:PaymentMethod) {}
	  
	  checkout(amount:number):void {
	  this.paymentMethod.pay(amount);
	  }
	  }
	  
	  ```
	  
	  ---
	- ### Usage (IMPORTANT)
	  
	  ```tsx
	  const order1 =newOrderService(newCardPayment());
	  order1.checkout(100);
	  
	  const order2 =newOrderService(newUpiPayment());
	  order2.checkout(200);
	  
	  ```
	- ### Why this follows DIP
	- High-level (`OrderService`) depends on `PaymentMethod`
	- Low-level (`CardPayment`, `UpiPayment`) also depend on same abstraction
	- No concrete dependency inside `OrderService`
	  
	  ---
	- ## How DIP appears in LLD problems
	- ### Parking Lot
	- `ParkingLot` depends on `PricingStrategy`
	- Not on `HourlyPricing` directly
	- ### Rate Limiter
	- Limiter depends on `RateLimitStore`
	- Not on Redis / In-memory store
	- ### Cache
	- Cache depends on `EvictionPolicy`
	- Not on LRU directly
	  
	  Interview phrase:
	  
	  > “I inverted dependencies by making both layers depend on abstractions.”
	  >
